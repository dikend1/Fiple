# Release 1.1 — distribution & App Store design (draft)

Status: draft — approved in-session by the owner 2026-07-13; formal doc acceptance
still human-only per repo governance. Target: everything on `feat/remote-terminal`
ships as Fiple 1.1. Context: 1.0 (Shortcuts removed) is live in the App Store for
both platforms.

## Idea

One codebase, three artifacts:

| Artifact | Channel | Contents |
| --- | --- | --- |
| FipleiOS 1.1.0 | App Store | Smart Trash review, Send to Mac + share extension, Fiple Pro paywall, onboarding, **terminal client** |
| FipleMac 1.1.0 (MAS) | Mac App Store | Same minus terminal — sandboxed, `Fiple.entitlements`; terminal hidden by the existing runtime check (`TerminalController.isFeatureAvailable`) |
| FipleMac 1.1.0 (Developer ID) | fiple.app/download | Full version with terminal; not sandboxed, hardened runtime, notarized |

Why the split: a real login shell cannot run inside the App Sandbox, so the
terminal host can never ship through the Mac App Store. An iOS *terminal client*
in the App Store is fine (Termius/Blink precedent) — the code executes on the
Mac, so guideline 2.5.2 is not implicated.

## Build configurations (chosen: variant A)

Today the posture is flipped by hand-editing `project.yml` — the standing risk is
archiving the dev (non-sandboxed) config for MAS. Replace with three named
configurations plus dedicated archive schemes, all in `project.yml` so they
survive `xcodegen generate`:

| Config | Sandbox | Entitlements | Team | Purpose |
| --- | --- | --- | --- | --- |
| `Debug` | off | `FipleTerminalDev.entitlements` | M3UQXL2W5B (automatic) | Day-to-day dev incl. local terminal — current workflow unchanged |
| `Release-MAS` | on | `Fiple.entitlements` | M3UQXL2W5B | Mac App Store archive |
| `Release-DevID` | off | `FipleTerminalDev.entitlements` + `ENABLE_HARDENED_RUNTIME: YES` | M3UQXL2W5B (Developer ID Application cert) | Notarized site download |

Schemes: `FipleMac-MAS` archives `Release-MAS`; `FipleMac-DevID` archives
`Release-DevID`; the plain `FipleMac` scheme stays on Debug. iOS keeps a single
release config (App Store only). Version bumps to `MARKETING_VERSION 1.1.0`,
`CURRENT_PROJECT_VERSION 6`.

Rejected alternatives: a second `FipleMacDevID` target (duplicates settings, two
products with one bundle id); keeping the manual flip + checklist (the risk this
design exists to remove).

## iOS: terminal entry when the Mac can't host it

Today the Tools card just greys out when no Mac advertises a terminal
(`ToolsView.swift`) — a reviewer paired with the MAS Mac (or nothing) sees a
permanently dead feature (guideline 2.1 risk). Change: the card stays tappable;
without an advertised terminal it opens a short explainer — the terminal needs
the full Fiple for Mac from fiple.app — with a link. Add
`FipleLinks.download = https://fiple.app/download`. This is simultaneously the
UX, the answer to App Review, and the funnel to the Developer ID build.

## Developer ID pipeline

1. Create a **Developer ID Application** certificate in team M3UQXL2W5B —
   Account Holder only can generate it (no such identity is in the local
   keychain today; only two Apple Development certs).
2. Archive via `FipleMac-DevID` → `xcrun notarytool submit` (App Store Connect
   API key or app-specific password) → `xcrun stapler staple` → zip/DMG.
3. Publish at `fiple.app/download`. Do not link the off-store download from MAS
   metadata; linking from the iOS in-app explainer and the website is fine.

## Submission package

- New screenshots (Smart Trash deck, Terminal, Send to Mac), What's New text.
- Review notes + demo video of the full iPhone↔Mac flow — the reviewer cannot
  pair the two apps themselves. Explain explicitly that the terminal requires
  the Mac companion and is hidden otherwise.
- Pre-archive gate: `swift test` in FipleKit green; both Mac configs and the iOS
  config build clean.

## Order of work

1. `project.yml` configs + schemes + version bump (code).
2. iOS terminal explainer + `FipleLinks.download` (code).
3. Human doc acceptance, then merge `feat/remote-terminal` → `main`.
4. Developer ID cert → notarization dry run → fiple.app/download page.
5. ASC: attach the two existing IAPs to the 1.1 version, sandbox-purchase
   check, agreement check, privacy label, metadata.
6. Archives (iOS App Store, Mac MAS, Mac DevID) → submit 1.1 + IAPs for review.

## Governance

BRD/PRD/ADR-0003 remain draft; acceptance is human-only and gates step 3. The
"never push feat/remote-terminal to main" rule was conditioned on Developer ID +
1.0 approval — 1.0 is approved, and this design supplies the Developer ID path,
so the merge happens in step 3 with the owner's explicit go.
