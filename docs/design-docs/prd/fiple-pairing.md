## Overview

This PRD covers how an iPhone and a Mac establish and maintain a trusted link,
from the user's perspective. It deliberately excludes tile content and execution
(see `prd/fiple-remote-tiles.md`). The guiding constraint from the BRD: **no
account, no password, no cloud, no list of nearby devices** — just a code.

---

## User-Observable Properties

- **Code-only entry** — On the iPhone the user sees a single field to type a
  4-digit code. There is never a browsable list of "Macs nearby".
- **One-time setup** — After a successful pairing, the phone remembers the Mac
  and reconnects automatically on later launches without re-entering the code.
- **Same-Wi-Fi requirement** — Pairing and control work only while both devices
  are on the same Wi-Fi network. The phone clearly signals when it is not
  connected.
- **Explicit disconnect** — The user can disconnect from the phone or the Mac.
  After disconnecting, reconnecting requires a **new** code generated by the Mac.
- **Honest connection state** — The phone always shows whether it is connected;
  it never silently queues taps against a dead link.

---

## Core Flow

1. On the Mac menu-bar app, the user clicks "Connect"; the Mac displays a
   4-digit code and begins advertising on the local network.
2. On the iPhone, the user types the code into the single code field.
3. The phone silently discovers Fiple Macs on the same Wi-Fi in the background;
   the code selects the correct Mac and authenticates the link.
4. On success, both devices show "connected" and the pairing is persisted on the
   phone.
5. On next launch, the phone reconnects automatically to the remembered Mac.
6. If the user taps "Disconnect", the link drops; the next connection requires a
   newly generated code.

---

## Success Criteria

- [ ] First-time pairing completes in under 30 seconds on the same Wi-Fi.
- [ ] No device list is ever shown; only a code field.
- [ ] A previously paired phone reconnects with zero taps on next launch.
- [ ] A wrong or expired code produces a clear, actionable message and the code
  can be regenerated.
- [ ] When Wi-Fi drops or devices are on different networks, the phone shows
  "not connected" rather than failing silently.

---

## Constraints

- Local network only; no cloud relay, no account system in MVP.
- Exactly one active phone↔Mac pairing per Mac at a time in MVP.
- Pairing secret must not be guessable by brute-forcing the 4-digit code from
  off-network actors (code is scoped to a short-lived, on-network handshake).

---

## Out Of Scope

- Cross-network / internet pairing.
- Multiple simultaneous phones controlling one Mac.
- Pairing with non-Apple devices.
